# 第三章 系统需求分析

需求分析是软件开发过程中的关键环节，直接决定了系统的功能范围和技术方向。本章对商城智能购物助手系统进行详细的需求分析，包括系统概述、用户角色分析、功能需求分析、非功能需求分析以及典型用例分析。

## 3.1 系统概述

### 3.1.1 系统目标

本系统旨在构建一个功能完善的网上商城平台，并在此基础上集成基于大语言模型的智能购物助手功能。系统的核心目标包括：

**（1）提供完整的电商购物体验**

系统需要实现网上商城的核心功能，包括用户注册登录、商品浏览搜索、购物车管理、订单创建支付、收货地址管理等，为用户提供完整的在线购物体验。

**（2）实现智能化的购物辅助**

通过集成大语言模型技术，系统提供智能购物助手功能，能够理解用户的自然语言需求，提供个性化的商品推荐和购物建议，提升用户的购物效率和满意度。

**（3）确保系统的安全可靠**

系统需要保障用户数据的安全，实现可靠的身份认证机制，防止未授权访问和数据泄露，确保交易过程的安全性。

## 3.3 功能需求分析

### 3.3.1 用户认证模块

用户认证模块负责用户的身份管理，包括注册、登录、信息维护等功能。

**（1）用户注册**

- 用户可以通过用户名和密码注册账户
- 用户名需要唯一，不能与已有用户重复
- 密码需满足最小长度要求（不少于6位）
- 注册成功后自动创建用户账户

**（2）用户登录**

- 用户使用用户名和密码进行登录
- 登录成功返回JWT Token
- Token有效期为24小时
- 支持记住登录状态

**（3）身份验证**

- 需要认证的接口必须携带有效Token
- Token无效或过期时返回401错误
- 前端检测到401错误后跳转登录页

**（4）用户信息管理**

- 用户可查看个人信息
- 支持修改头像、手机、邮箱等信息
- 支持修改密码（需验证原密码）

### 3.3.2 商品管理模块

商品管理模块负责商品信息的展示和管理。

**（1）商品列表展示**

- 分页展示商品列表
- 支持按分类筛选商品
- 支持按价格、销量等排序
- 显示商品图片、名称、价格等基本信息

**（2）商品详情查看**

- 展示商品详细信息
- 包括商品图片、描述、规格、库存等
- 显示商品价格和优惠信息
- 提供加入购物车和立即购买入口

**（3）商品搜索**

- 支持按商品名称关键词搜索
- 支持在搜索结果中筛选和排序
- 高亮显示匹配的关键词

**（4）商品管理（管理员）**

- 添加新商品
- 编辑商品信息
- 上下架商品
- 管理商品分类

### 3.3.3 购物车模块

购物车模块管理用户的待购商品。

**（1）添加到购物车**

- 用户可将商品添加到购物车
- 可选择商品数量
- 同一商品再次添加时累加数量
- 校验库存是否充足

**（2）购物车列表**

- 显示购物车中的所有商品
- 展示商品图片、名称、单价、数量、小计
- 显示购物车商品总数和总金额
- 自动同步商品价格和状态变化

**（3）修改商品数量**

- 支持增加或减少商品数量
- 数量不能小于1
- 数量不能超过库存
- 实时更新小计和总金额

**（4）删除商品**

- 支持删除单个商品
- 支持清空购物车
- 删除前进行确认提示

### 3.3.4 订单管理模块

订单管理模块处理用户的购买订单。

**（1）创建订单**

- 从购物车选择商品结算
- 选择收货地址
- 生成唯一订单编号
- 计算订单总金额
- 扣减商品库存
- 清空已结算的购物车商品

**（2）订单列表**

- 分页显示用户的历史订单
- 支持按订单状态筛选
- 显示订单编号、金额、状态、时间等
- 提供订单详情入口

**（3）订单详情**

- 显示订单基本信息
- 显示订单商品列表
- 显示收货地址信息
- 显示订单状态和时间线

### 3.3.7 运营与维护需求

系统不仅要实现功能，还需要满足运营与维护方面的需求。具体包括：

- 日志记录：记录关键操作、异常信息和对话历史，便于后续排查和统计分析。
- 监控报警：对系统性能、请求错误率、数据库连接等关键指标进行监控，并在出现异常时发送告警。
- 配置化管理：将可变参数如JWT过期时间、LLM供应商、缓存策略等配置外部化，方便上线后调整。
- 可部署性：支持本地环境、测试环境和生产环境的差异化配置，便于系统发布和维护。

这些运营与维护需求能够提高系统的可观测性和可运维性，提升长期运行稳定性。

## 3.4 非功能需求分析

### 3.4.1 性能需求

**表3-3 系统性能需求**

| 指标 | 要求 | 说明 |
|------|------|------|
| 页面加载时间 | ≤ 3秒 | 首页、列表页等主要页面 |
| API响应时间 | ≤ 500ms | 普通CRUD接口 |
| 智能助手首字响应 | ≤ 2秒 | SSE流式输出首个token |
| 并发用户数 | ≥ 100 | 同时在线用户数 |
| 数据库查询 | ≤ 100ms | 单表查询响应时间 |

### 3.4.2 安全需求

**（1）身份认证安全**

- 使用JWT进行身份认证
- Token有效期合理设置
- 密码使用BCrypt加密存储
- 敏感接口必须验证身份

**（2）数据传输安全**

- 支持HTTPS协议（生产环境）
- 防止SQL注入攻击
- 防止XSS跨站脚本攻击
- 防止CSRF跨站请求伪造

**（3）数据存储安全**

- 密码等敏感信息加密存储
- 定期备份数据库
- 日志记录关键操作

### 3.4.3 可用性需求

**（1）系统可用性**

- 系统可用率 ≥ 99%
- 支持优雅降级
- 具备故障恢复能力

**（2）用户体验**

- 界面简洁美观
- 操作流程直观
- 错误提示友好
- 支持响应式布局

### 3.4.4 可扩展性需求

**（1）功能扩展性**

- 模块化设计，便于添加新功能
- LLM服务可切换不同提供商
- RAG数据源可扩展

**（2）性能扩展性**

- 支持水平扩展
- 数据库可主从分离
- 支持缓存层扩展

### 3.4.5 可维护性需求

- 代码规范，注释完善
- 日志记录完整
- 配置外部化，便于修改
- 提供API文档

### 3.4.6 可靠性需求

- 服务可恢复性：系统异常或重启后能快速恢复服务，不影响用户业务流程。
- 数据一致性：重要业务操作（如订单创建、库存扣减）必须在事务中完成，避免出现数据不一致情况。
- 高可用性：关键服务节点应支持快速切换和容错机制，降低单点故障风险。
- 容错处理：对外部服务调用（如LLM API）要实现重试和降级策略，确保系统在外部依赖异常时仍能提供基础服务。

### 3.4.7 可审计性需求

- 日志记录：对用户登录、订单支付、商品修改、助手对话等关键操作进行详细日志记录。
- 审计追踪：记录操作时间、用户ID、结果状态和异常信息，便于事后审计与问题定位。
- 数据访问控制：对敏感数据访问进行权限校验和访问日志记录，保证业务数据安全。
- 变更管理：关键配置和业务规则变更应保留历史记录，支持回滚与责任追踪。

这些可审计性需求可以为系统的长期运营、故障排查和安全合规提供重要保障。

## 3.5 典型用例分析

### 3.5.1 用例1：用户购物流程

**用例名称**：完整购物流程

**参与者**：普通用户

**前置条件**：用户已注册并登录

**基本流程**：
1. 用户浏览商品列表或使用搜索功能
2. 用户查看感兴趣商品的详情
3. 用户将商品添加到购物车
4. 用户进入购物车查看已选商品
5. 用户调整商品数量
6. 用户点击结算，选择收货地址
7. 系统创建订单，显示订单详情
8. 用户点击支付，完成支付
9. 系统更新订单状态，显示支付成功

**备选流程**：
- 3a. 库存不足：系统提示库存不足，无法添加
- 6a. 无收货地址：引导用户添加收货地址
- 8a. 取消支付：订单保持待支付状态

### 3.5.2 用例2：智能助手咨询

**用例名称**：智能助手购物咨询

**参与者**：普通用户

**前置条件**：用户已登录

**基本流程**：
1. 用户打开智能助手页面
2. 系统显示会话列表和对话区域
3. 用户输入购物相关问题（如"我想买一款适合送女朋友的手机"）
4. 系统通过RAG检索相关商品
5. 系统调用LLM生成回复
6. 系统通过SSE流式输出回复内容
7. 系统在回复后展示推荐商品卡片
8. 用户点击商品卡片查看详情
9. 用户可继续追问或开始新话题

**备选流程**：
- 4a. 未检索到相关商品：助手基于通用知识回答
- 5a. LLM服务异常：提示用户稍后重试

### 3.5.3 用例3：订单管理

**用例名称**：查看和管理订单

**参与者**：普通用户

**前置条件**：用户已登录且有历史订单

**基本流程**：
1. 用户进入"我的订单"页面
2. 系统显示订单列表，按时间倒序排列
3. 用户可以按状态筛选订单
4. 用户点击某订单查看详情
5. 系统显示订单商品、金额、状态、地址等信息
6. 对于待支付订单，用户可以支付或取消
7. 支付/取消后，系统更新订单状态

## 3.6 本章小结

本章对商城智能购物助手系统进行了详细的需求分析。首先明确了系统的目标和边界，然后识别了系统的三类用户角色（普通用户、管理员、游客）及其权限。在功能需求方面，详细分析了用户认证、商品管理、购物车、订单管理、收货地址、智能助手等六大功能模块的具体需求，并绘制了用例图、活动图、状态图等UML图表。在非功能需求方面，从性能、安全、可用性、可扩展性、可维护性五个维度提出了具体要求。最后，通过三个典型用例的分析，展示了系统的主要使用场景。本章的需求分析成果为后续的系统设计和实现提供了明确的目标和依据。

通过对功能需求、非功能需求和典型用例的深入分析，本章进一步明确了系统的核心业务场景和用户体验要求。对于后续系统设计而言，这些需求分析结果将直接指导模块划分、接口定义和流程设计，确保最终实现满足用户预期。

同时，本章还对系统需求的风险进行了初步评估。智能助手模块的数据依赖、LLM服务可用性以及多轮对话上下文管理是关键风险点。针对这些风险，后续设计与实现中需要通过容错机制、异步队列、日志监控和回退策略来保障系统稳定性，并逐步迭代优化用户体验。

此外，还需重视需求变更管理和技术债务控制。在开发过程中，需求变更应通过评审流程进行合理评估，避免频繁调整导致设计不稳定。对于可能的技术债务，应在后续迭代中安排专项重构与优化，保证系统长期演进具有较高的可维护性。
