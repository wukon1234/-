# 第五章 系统实现

本章详细介绍商城智能购物助手系统的具体实现过程，包括开发环境配置、核心模块的实现、关键技术的应用以及系统界面展示等内容。系统采用前后端分离的架构设计，后端基于Spring Boot框架构建，前端基于Vue 3框架构建，通过RESTful API进行数据交互，智能助手模块采用RAG技术和LLM服务实现智能问答功能。

## 5.1 开发环境配置

### 5.1.1 后端环境配置

**（1）项目依赖配置**

后端项目使用Maven进行依赖管理，通过在pom.xml文件中声明项目依赖，Maven会自动下载和管理所需的JAR包。核心依赖及其作用说明如下：

Spring Boot Web依赖提供了完整的Web服务能力，包括RESTful API的构建、请求处理、响应封装等功能。该依赖内置了Tomcat服务器，无需额外配置应用容器即可直接运行Web应用。

MyBatis Plus依赖是MyBatis的增强工具，提供了更便捷的数据库操作能力。该依赖封装了常见的CRUD操作，支持Lambda表达式查询、条件构造器、 分页插件等功能，大大简化了数据库操作的代码编写量。在本系统中，MyBatis Plus版本指定为3.5.3，以确保与Spring Boot 2.7.x版本的兼容性。

MySQL驱动依赖用于连接MySQL数据库，JDBC驱动负责建立Java应用程序与MySQL服务器之间的连接通道。MySQL Connector/J是MySQL官方提供的JDBC驱动实现，支持数据库连接池和连接管理功能。

JWT（JSON Web Token）依赖用于生成和验证JSON格式的令牌。jjwt库提供了JWT的签发、验证和解析功能，系统通过JWT实现用户身份认证和会话管理。在本系统中，JWT版本指定为0.9.1，该版本与Spring Boot 2.x系列具有较好的兼容性。

Lombok依赖是一个Java代码生成工具，通过注解自动生成Getter、Setter、构造器、Builder模式等代码，减少样板代码的编写。在本系统中，Lombok主要用于实体类的定义，配合IDE插件使用可以显著提升开发效率。

**（2）应用配置文件**

系统使用application.yml作为应用配置文件，采用YAML格式编写，相比传统的properties文件具有更好的可读性和层次结构。配置文件主要包含以下配置项：

服务器端口配置指定系统运行时的HTTP服务端口，本系统配置为8080端口。前端开发服务器通过代理将API请求转发至该端口，实现前后端联调。

数据库连接配置包括连接URL、用户名、密码和驱动类名。连接URL指定了数据库的地址、端口和数据库名称，本系统使用的数据库名称为mall_assistant。用户名和密码用于数据库身份验证，确保应用具有正确的数据库访问权限。驱动类名指定了JDBC使用的数据库驱动实现。

MyBatis Plus配置指定了Mapper接口的XML映射文件位置为classpath:mapper/*.xml，并开启了下划线命名转驼峰命名的自动转换功能。这一配置使得数据库字段（如user_name）可以自动映射到Java实体类属性（如userName），简化了属性映射的配置工作。

JWT配置包括密钥secret和过期时间expiration两个关键配置项。密钥用于JWT令牌的签名验证，过期时间以毫秒为单位指定，本系统配置为86400000毫秒，即24小时。生产环境中密钥应使用复杂的随机字符串，并妥善保管以防止泄露。

LLM服务配置指定了大语言模型服务提供商为qwen（通义千问）以及API密钥。API密钥是调用通义千问服务的凭证，需要从阿里云百炼平台申请获取。

### 5.1.2 前端环境配置

**（1）项目初始化**

前端项目使用Vite作为构建工具，Vite是新一代的前端构建工具，相比Webpack具有更快的冷启动和热更新速度。项目创建时指定使用Vue 3和TypeScript模板，生成的项目结构包含src目录（源代码）、public目录（静态资源）、index.html（入口HTML）等标准目录和文件。

技术栈方面，Vue 3是当前最新的Vue主版本，采用了Composition API和响应式系统的重写，提供了更好的TypeScript支持和性能表现。TypeScript是JavaScript的超集，提供了静态类型检查和面向对象编程能力，能够在开发阶段发现潜在的类型错误，提高代码质量。Element Plus是基于Vue 3的UI组件库，提供了丰富的企业级UI组件，包括按钮、表单、表格、对话框等常用组件，能够快速构建美观的用户界面。

项目依赖安装完成后，需要额外安装Element Plus（UI组件库）、Pinia（状态管理库）、Axios（HTTP请求库）和Vue Router（路由管理库）。Pinia是Vue 3推荐的状态管理库，相比Vuex具有更简单的API和更好的TypeScript支持。Axios用于封装HTTP请求，Vue Router用于管理前端路由，实现单页应用的页面切换功能。

开发环境建议使用Node.js 16或更高版本，以支持ES Modules和现代JavaScript语法。包管理工具可以选择npm或pnpm，两者都能有效管理项目依赖。保持npm或pnpm版本与项目依赖要求一致，可以避免因工具版本差异导致的安装失败或构建异常。

在团队开发过程中，建议使用统一的代码风格和提交规范。ESLint用于检查代码语法和风格问题，Prettier用于代码格式化，两者配合使用可以确保团队成员的代码风格一致。Git commit hooks可以在提交前自动执行代码检查，避免不符合规范的代码进入版本库。

**（2）Vite配置**

前端项目通过vite.config.ts文件进行构建配置，该文件使用TypeScript编写，导出一个包含各种配置项的对象。

插件配置项指定了项目使用的Vite插件列表，vue()插件是开发Vue 3项目必需的配置，它负责处理.vue文件的解析和模板编译。

开发服务器配置包括端口号和代理设置。端口号配置为5173，这是Vite开发服务器的默认端口。代理配置将/api路径的请求转发至后端服务器，该配置解决了开发阶段的跨域问题，使得前端可以顺利调用后端API。配置中的target指定了后端服务的地址，changeOrigin设置为true可以修改请求头中的Origin字段，确保后端能够正确处理跨域请求。

路径别名配置将@符号映射到/src目录，简化了模块导入路径的书写。例如，原本需要写import xxx from '../../components/xxx'的导入语句，可以简写为import xxx from '@/components/xxx'，提高了代码的可读性和维护性。

在实际部署前，建议通过环境变量区分开发、测试和生产环境。前端可以使用.env.development和.env.production文件分别配置不同环境的参数，如API地址、日志级别和第三方服务开关等。这样当系统上线时，只需切换构建环境即可使用正确的配置，避免硬编码带来的配置管理和安全风险。

### 5.1.3 数据库初始化

系统通过执行SQL初始化脚本创建数据库和表结构。首先创建名为mall_assistant的数据库，指定字符集为utf8mb4（支持存储表情符号等四字节字符）和排序规则为utf8mb4_unicode_ci（基于Unicode编码的排序规则）。

用户表（user）用于存储用户基本信息，包含用户ID（主键自增）、用户名（唯一约束）、密码（加密存储）、邮箱、电话、头像、角色和状态等字段。用户名唯一约束确保不会出现重复用户名，角色字段用于区分普通用户和管理员，状态字段用于控制账户是否可用。

商品表（product）用于存储商品信息，包含商品ID（主键自增）、商品名称、描述、分类ID、价格（精确到小数点后两位）、库存、图片URL和状态等字段。价格字段使用DECIMAL类型确保金额计算的精确性，库存字段记录当前商品的可用数量，状态字段用于控制商品是否上架销售。

其他表结构还包括分类表、购物车表、订单表、订单项表、收货地址表、会话表和消息表等，这些表结构共同构成了系统的数据存储层，为各项业务功能的实现提供了数据支持。

## 5.2 用户认证模块实现

### 5.2.1 JWT工具类实现

系统实现了JwtUtil工具类，作为JWT令牌管理的核心组件。该工具类被Spring框架管理为一个Bean组件，可以在其他组件中通过依赖注入的方式使用。

JwtUtil工具类的主要功能包括三个方面：令牌生成、令牌解析和令牌验证。

令牌生成功能通过generateToken方法实现，该方法接收用户ID作为参数，使用JJWT库提供的方法构建JWT令牌。令牌的生成过程包括设置主题（用户ID）、签发时间、过期时间和签名算法。系统采用HS512算法进行签名，该算法具有较高的安全性，能够有效防止令牌被篡改。签名时使用的密钥从配置文件中读取，确保密钥的安全性和可配置性。

令牌解析功能通过getUserIdFromToken方法实现，该方法接收令牌字符串作为参数，使用JJWT库的方法解析令牌并提取其中的用户ID。解析过程中会验证签名的有效性，如果签名不合法会抛出异常。成功解析后返回令牌中包含的用户ID，供后续业务处理使用。

令牌验证功能通过validateToken方法实现，该方法接收令牌字符串作为参数，通过尝试解析令牌来验证其有效性。如果令牌格式正确、签名有效且未过期，返回true；否则返回false。该方法主要用于拦截器验证请求中的令牌，以及登录时验证令牌的有效性。

工具类通过@Value注解从配置文件读取jwt.secret和jwt.expiration两个配置项，实现了配置与代码的分离。这种设计使得密钥和过期时间可以独立修改，无需重新编译代码，提高了系统的可维护性和安全性。

### 5.2.2 JWT拦截器实现

系统实现了JwtInterceptor拦截器，用于验证请求中的JWT令牌，实现用户身份认证功能。拦截器实现了Spring MVC的HandlerInterceptor接口，在请求处理之前执行认证检查。

拦截器的验证流程如下：首先检查请求方法是否为OPTIONS请求，如果是则直接放行，因为浏览器的预检请求通常使用OPTIONS方法。然后从请求头中获取Authorization字段，检查其值是否以"Bearer "开头，如果不是则返回401未授权状态。接着提取令牌字符串，调用JwtUtil的validateToken方法验证令牌有效性，如果令牌无效同样返回401状态。最后从有效令牌中提取用户ID，将其存入请求属性中，供后续的处理方法使用。

拦截器验证失败时会向响应写入JSON格式的错误信息，包含状态码和错误消息。前端可以通过解析响应JSON获取错误原因，并提示用户进行相应的操作，如重新登录等。

拦截器配置在Spring MVC的注册配置中生效，可以指定需要拦截和需要放行的路径。通常需要放行的路径包括登录接口、注册接口和静态资源路径，而其他API接口都需要经过身份验证。

### 5.2.3 用户服务实现

系统实现了UserServiceImpl类，提供用户注册和登录功能的业务逻辑。该类继承自MyBatis Plus的ServiceImpl类，自动获得了基本的CRUD操作能力，并实现了自定义的UserService接口。

用户注册功能通过register方法实现，接收用户名和密码作为参数。注册流程首先检查用户名是否已存在，调用MyBatis Plus的条件查询方法查询数据库中是否存在同名用户，如果存在则返回错误信息。如果用户名可用，则创建新的用户对象，设置用户名、加密后的密码、默认角色和状态，然后保存到数据库。密码加密使用BCryptPasswordEncoder实现，该编码器采用BCrypt哈希算法，能够有效防止彩虹表攻击，安全性较高。注册成功后返回成功信息。

用户登录功能通过login方法实现，接收用户名和密码作为参数。登录流程首先查询用户名对应的用户记录，如果不存在则返回用户不存在的错误信息。如果用户存在，则使用BCryptPasswordEncoder验证密码是否匹配，如果不匹配则返回密码错误的错误信息。密码验证通过后，调用JwtUtil的generateToken方法生成JWT令牌，并将令牌和用户信息一起返回给前端。前端获取令牌后通常会存储在localStorage中，后续请求时携带在请求头中用于身份验证。

用户服务还提供了根据用户ID查询用户信息、修改用户资料等辅助功能，这些功能通过继承自ServiceImpl的方法或自定义查询方法实现。

## 5.3 商品模块实现

### 5.3.1 商品列表查询

系统实现了ProductController类，提供商品列表查询和商品详情获取两个核心接口。控制器使用@RestController注解标记为RESTful控制器，所有返回值会自动序列化为JSON格式。

商品列表查询接口通过list方法实现，支持分页查询、分类筛选和关键词搜索三个维度的查询条件。分页参数包括page（当前页码，默认1）和size（每页数量，默认10），系统会根据这两个参数计算分页偏移量。分类筛选参数categoryId为可选参数，当提供该参数时，查询结果只会包含指定分类的商品。关键词搜索参数keyword同样为可选参数，当提供该参数时，查询结果只会包含商品名称包含关键词的商品。

查询条件通过MyBatis Plus的LambdaQueryWrapper构建，首先添加状态条件确保只查询上架状态的商品，然后根据是否提供分类ID和关键词添加相应的筛选条件。查询结果按创建时间倒序排列，使得最新上架的商品排在前面。最后调用分页查询方法返回分页结果，包括当前页数据、总记录数、总页数等信息。

商品详情接口通过detail方法实现，接收商品ID作为路径参数。查询时首先尝试获取指定ID的商品记录，如果商品不存在则返回错误信息。商品存在则返回完整的商品信息，包括名称、描述、价格、库存、图片等详细信息。

### 5.3.2 商品搜索实现

系统实现了ProductServiceImpl类，提供商品搜索和分类查询功能。搜索功能支持对商品名称和描述进行模糊匹配，能够根据用户输入的关键词查找相关商品。

搜索功能通过searchProducts方法实现，接收关键词和返回数量限制两个参数。查询条件使用MyBatis Plus的条件构造器构建，首先筛选上架状态的商品，然后使用and方法组合多个like条件，实现对商品名称和描述的模糊匹配。查询结果通过last方法添加数量限制，确保只返回指定数量的搜索结果，避免返回过多无用的数据。

分类查询功能通过getByCategory方法实现，接收分类ID作为参数。查询条件筛选指定分类且上架状态的商品，返回该分类下的所有商品列表。该功能用于商品分类浏览场景，用户选择某个分类后可以查看该分类下的所有商品。

搜索和分类查询功能为智能助手模块的RAG检索提供了基础支持，智能助手可以根据用户的问题提取关键词，调用这些搜索功能查找相关商品，为后续的商品推荐提供数据支持。

## 5.4 购物车模块实现

### 5.4.1 添加购物车

系统实现了CartController类，处理购物车相关的HTTP请求。控制器提供了添加商品到购物车和获取购物车列表两个核心接口。

添加购物车接口通过add方法实现，接收包含商品ID和数量的请求体参数。方法首先从请求属性中获取当前登录用户的ID，该用户ID由JWT拦截器在请求处理前存入。然后调用购物车服务的addToCart方法处理业务逻辑，将用户ID、商品ID和数量传递给服务层。

获取购物车列表接口通过list方法实现，从请求属性中获取用户ID后调用购物车服务的getCartList方法获取该用户的购物车商品列表。返回的数据包含每件商品的详细信息，如商品图片、名称、单价、购买数量和小计金额等，方便前端展示购物车内容。

### 5.4.2 购物车服务实现

系统实现了CartServiceImpl类，处理购物车的业务逻辑。该类继承自ServiceImpl，拥有基本的数据库操作能力，并实现了CartService接口定义的方法。

添加商品到购物车的功能通过addToCart方法实现，该方法使用@Transactional注解标记为事务方法，确保操作的数据一致性。添加流程首先检查商品是否存在且处于上架状态，如果商品不存在或已下架则返回错误信息。然后检查库存是否充足，如果购买数量超过库存则返回错误信息。

接着检查该商品是否已在用户购物车中，如果存在则更新数量（累加新的购买数量），更新时再次检查累加后的数量是否超过库存。更新操作会修改现有购物车记录的quantity字段。如果不存在则创建新的购物车记录，保存用户ID、商品ID和数量等信息。

购物车服务的getCartList方法用于获取用户的购物车列表，该方法调用Mapper的自定义查询方法，关联查询购物车表和商品表，返回购物车商品视图对象列表。视图对象包含购物车ID、商品信息、单价、数量和小计金额等字段，方便前端直接展示。

购物车模块的实现确保了用户可以方便地管理购物车商品，支持修改购买数量和删除商品等操作，为用户提供了流畅的购物体验。

## 5.5 订单模块实现

### 5.5.1 创建订单

系统实现了OrderController类，处理订单相关的HTTP请求。控制器提供了创建订单和支付订单两个核心接口。

创建订单接口通过create方法实现，接收包含收货地址ID和购物车ID列表的请求体参数。方法从请求属性中获取当前登录用户的ID，然后调用订单服务的createOrder方法处理订单创建的业务逻辑。

支付订单接口通过pay方法实现，接收订单ID作为路径参数。该接口用于模拟订单支付流程，在实际系统中可以对接第三方支付平台实现真实的支付功能。支付成功后会将订单状态更新为已支付，并记录支付时间等信息。

### 5.5.2 订单服务实现

系统实现了OrderServiceImpl类，处理订单创建的完整业务流程。该类使用@Transactional注解标记事务，确保订单创建过程中的多个数据库操作要么全部成功，要么全部回滚，保持数据一致性。

订单创建流程包含以下步骤：

第一步，查询购物车商品。根据传入的购物车ID列表查询对应的购物车记录，如果购物车为空则返回错误信息。

第二步，验证库存并计算总金额。遍历购物车中的每件商品，检查库存是否充足。如果库存不足则返回错误信息，并指明是哪种商品库存不足。库存充足则计算该商品的金额（单价乘以数量），累加到总金额中，同时创建订单项对象记录商品信息。

第三步，创建订单记录。生成唯一的订单号（格式为"ORD"加上时间戳和随机数），设置订单的用户ID、总金额、收货地址ID和订单状态（初始为待支付）等信息，保存订单记录到数据库。

第四步，创建订单项记录。遍历之前创建的订单项对象列表，为每个订单项设置所属订单的ID，然后插入到数据库。订单项记录了每个商品的名称、单价和数量等信息，用于后续的订单详情展示。

第五步，扣减库存。遍历购物车商品，调用商品服务的库存扣减方法，减少相应商品的库存数量。库存扣减是订单创建的关键步骤，必须确保在事务中完成，避免超卖问题。

第六步，清空购物车。删除用户提交的购物车记录，完成整个订单创建流程。

该实现确保了订单创建的完整性和数据一致性，每个步骤都有相应的错误处理，任何一步失败都会触发事务回滚，保证不会出现库存扣减但订单未创建等异常情况。

## 5.6 智能助手模块实现

### 5.6.1 RAG检索服务

系统实现了RAGService接口和SimpleRAGServiceImpl实现类，提供检索增强生成（Retrieval-Augmented Generation）能力。RAG技术通过先检索相关文档再生成答案的方式，提高了LLM回答的准确性和相关性。

RAGService接口定义了检索相关商品和格式化商品信息两个方法。searchRelevantProducts方法接收用户查询和返回数量参数，提取用户查询中的关键词后搜索相关商品。formatProductsAsContext方法将商品列表格式化为文本上下文，作为LLM生成回复时的参考信息。

SimpleRAGServiceImpl实现了上述接口，具体实现采用基于关键词的检索策略。关键词提取功能通过extractKeywords方法实现，该方法首先定义停用词列表（如"我"、"想"、"要"、"买"等常见无意义词汇），然后对用户查询进行分词处理，过滤掉长度为1的字符和停用词列表中的词汇，保留有意义的关键词。

商品搜索功能构建复杂的查询条件，筛选上架状态的商品，并使用or条件组合多个关键词的like查询，实现对商品名称和描述的匹配。查询结果限制数量为指定的topK参数值，避免返回过多无关结果。

上下文格式化功能将搜索到的商品格式化为易于阅读的文本，每件商品包含序号、名称、价格和描述（截取前50个字符）。格式化后的文本作为用户消息的补充信息发送给LLM，使LLM能够基于真实的商品数据进行回答，提高回复的准确性和有用性。

### 5.6.2 LLM服务接口

系统定义了LLMService接口，提供与大语言模型交互的统一抽象。该接口采用策略模式设计，便于切换不同的LLM服务提供商。

LLMService接口定义了chat和chatStream两个方法。chat方法用于普通对话，接收消息列表并返回完整的回复字符串，适用于短文本生成场景。chatStream方法用于流式对话，接收消息列表和回调接口，边生成边返回回复内容，适用于长文本生成和实时交互场景。

流式回调接口StreamCallback是@FunctionalInterface类型的函数式接口，定义了onToken方法接收LLM返回的每个Token。这种设计使得LLM返回的内容可以即时处理和展示，无需等待完整响应生成完毕，显著提升了用户交互体验。

### 5.6.3 通义千问服务实现

系统实现了QwenLLMServiceImpl类，调用通义千问API实现LLM服务。该类使用@ConditionalOnProperty注解标记，仅当配置文件中llm.provider配置项值为qwen时才会被Spring容器加载，实现LLM服务提供商的灵活切换。

API调用功能通过chatStream方法实现，使用Java标准库的HttpClient发送HTTP请求。请求构建包括设置API地址、Authorization头（Bearer令牌认证）、Content-Type头和X-DashScope-SSE头（启用SSE流式输出），请求体采用JSON格式，包含模型名称、消息列表和输出参数等配置项。

流式响应处理通过HttpResponse.BodyHandlers.ofLines方法获取响应体的行迭代器，逐行解析SSE格式的数据。每行数据以"data:"开头，后面跟着JSON格式的响应内容。解析出Token后通过回调接口的onToken方法传递给调用方，调用方通常会将Token追加到响应内容中实现实时显示效果。

请求体构建功能通过buildRequestBody方法实现，将消息列表转换为通义千问API要求的JSON格式。消息类型包括system（系统消息，定义助手角色和行为）、user（用户消息）和assistant（助手消息，用于对话历史）。模型名称指定为qwen-turbo，该模型具有较好的性价比和响应速度，适合对话场景使用。

### 5.6.4 智能助手控制器

系统实现了AssistantController类，处理智能助手相关的HTTP请求。控制器提供了流式对话和获取会话列表两个核心接口。

流式对话接口通过chatStream方法实现，使用SSE（Server-Sent Events）技术实现服务器向客户端的持续推送。该接口创建SseEmitter对象作为SSE连接的核心，指定超时时间为60秒（60000毫秒），然后异步调用助手服务处理对话请求。

异步处理使用CompletableFuture的runAsync方法，在后台线程中执行耗时的对话处理逻辑。处理完成后通过SseEmitter发送完成信号，或在发生异常时发送错误信号。这种设计避免了阻塞HTTP请求线程，提高了服务器的并发处理能力。

获取会话列表接口通过getConversations方法实现，从请求属性获取用户ID后调用助手服务获取该用户的所有会话记录。会话记录包括会话ID、标题、创建时间和最后更新时间等信息，前端用于展示会话列表供用户选择。

### 5.6.5 智能助手服务实现

系统实现了AssistantServiceImpl类，处理智能助手的核心业务逻辑，是整个智能助手模块的核心组件。

对话处理功能通过handleStreamChat方法实现，包含完整的对话处理流程。

第一步，保存用户消息。将用户发送的消息内容保存到会话消息表中，记录消息角色为USER。这保证了对话历史的完整性，LLM可以基于上下文生成连贯的回复。

第二步，RAG检索相关商品。调用RAG服务的searchRelevantProducts方法，传入用户消息作为查询，获取最相关的5件商品。然后调用formatProductsAsContext方法将商品信息格式化为上下文文本。

第三步，获取历史消息。调用会话服务的getRecentMessages方法，获取当前会话最近10条消息记录，用于构建完整的对话上下文。

第四步，构建消息列表。创建一个包含系统消息、历史消息和当前用户消息的消息列表。系统消息定义了助手的角色定位，要求助手根据用户需求和商品信息给出专业的购物建议。

第五步，流式调用LLM。调用LLM服务的chatStream方法，传入消息列表和Token回调函数。回调函数负责将每个Token发送给客户端，并累加到完整响应字符串中。发送时使用SseEmitter.event()方法构建SSE事件，事件名称为"message"，数据为包含Token的JSON对象。

第六步，发送推荐商品。如果RAG检索到了相关商品，会将这些商品作为推荐结果发送给客户端。发送时使用事件名称为"products"的SSE事件，数据为商品列表的JSON数组。前端收到后会展示商品推荐卡片，方便用户查看和选择推荐商品。

第七步，保存助手回复。将LLM生成的完整回复保存到会话消息表中，记录消息角色为ASSISTANT，完成对话历史的记录。

第八步，发送完成信号。向客户端发送事件名称为"done"的SSE事件，通知客户端对话已完成，然后调用emitter.complete()关闭SSE连接。如果处理过程中发生异常，调用emitter.completeWithError(e)通知客户端连接异常关闭。

该实现确保了智能助手能够提供准确、相关、实时的购物建议，通过RAG技术结合真实的商品数据进行回复，通过SSE技术实现流式输出提升用户体验，通过消息历史管理实现多轮对话能力。

## 5.7 前端实现

### 5.7.1 Axios封装

前端实现了统一的HTTP请求处理模块request.ts，基于Axios库封装了项目的API调用逻辑。该模块提供了可复用的请求配置和统一的错误处理机制。

请求创建配置指定了baseURL为'/api'，表示所有请求都会发送到这个基础路径（实际请求会被Vite代理转发到后端服务器）。timeout设置为30000毫秒（30秒），用于处理网络异常或服务器响应超时的情况。

请求拦截器在每个请求发送前执行，用于添加认证信息和预处理请求配置。拦截器从localStorage中读取token，如果存在则在请求头的Authorization字段中添加Bearer令牌。这样后端拦截器就能验证请求的用户身份，实现接口的访问控制。

响应拦截器在收到服务器响应后执行，用于统一处理响应数据和错误情况。成功响应（HTTP状态码为200）会直接返回响应数据，供后续逻辑使用。如果响应表示业务错误（如后端返回code不为200），会显示错误提示并拒绝Promise。错误响应会根据状态码进行区分处理：401未授权会清除本地token并跳转到登录页，提示用户重新登录；其他错误会显示网络错误提示。

这种统一的请求封装使得项目中所有API调用都遵循相同的处理逻辑，简化了代码编写，提高了错误处理的一致性。

### 5.7.2 SSE客户端实现

前端实现了SSE客户端模块assistant.ts，用于接收智能助手的流式回复。该模块封装了SSE连接的建立、消息解析和回调处理逻辑。

流式对话函数streamChat接收会话ID、消息内容、Token回调函数、商品回调函数和完成回调函数五个参数。函数内部首先获取存储在本地的token，然后使用fetch API发送POST请求到流式对话接口，请求头中包含Content-Type和Authorization信息。

响应体的处理使用ReadableStream的getReader方法逐块读取数据。每块数据使用TextDecoder解码后按行分割，解析SSE格式的事件流。事件解析时首先识别event:行的内容确定事件类型，然后解析data:行的JSON数据。

对于message类型的消息事件，如果数据中包含token字段，调用onToken回调函数通知调用方。这样可以实现对话内容的实时显示，用户可以看到助手正在逐字生成回复。

对于products类型的消息事件，如果数据是数组格式，调用onProducts回调函数通知调用方。这样可以实现推荐商品的实时展示，用户可以在对话过程中看到助手推荐的商品。

最后当流式响应结束时，调用onComplete回调函数通知调用方对话已完成，触发界面状态的更新。

### 5.7.3 智能助手页面组件

前端实现了智能助手页面组件Assistant.vue，包含完整的对话交互界面。该组件采用Vue 3的Composition API和TypeScript编写，具有清晰的逻辑结构和良好的类型支持。

组件的模板部分包含会话列表区域和对话区域两个主要部分。会话列表区域使用ConversationList子组件，传入会话列表和当前选中会话ID，监听选择和创建会话事件。对话区域使用ChatWindow子组件显示消息列表和加载状态，使用MessageInput子组件提供消息输入功能。

组件的状态管理使用ref函数创建响应式状态，包括会话列表、当前选中会话ID、消息列表、输入内容和加载状态等。这些状态用于驱动界面的渲染和交互逻辑。

发送消息的逻辑首先检查输入内容是否为空或是否处于加载状态，如果为空或正在加载则直接返回。然后将用户输入添加到消息列表中显示，同时添加一个空的助手消息占位符用于后续内容追加。接着调用streamChat函数发起流式对话请求，Token回调用于追加助手消息内容，商品回调用于更新助手消息的推荐商品，完成回调用于结束加载状态。

组件在挂载时调用getConversations接口获取会话列表，加载用户的历史会话数据，方便用户继续之前的对话。

这种组件化的设计使得代码结构清晰，职责分明，便于维护和扩展。消息的实时更新和推荐商品的展示为用户提供了直观、友好的交互体验。

## 5.8 系统界面展示

### 5.8.1 主要界面说明

**（1）商城首页**

商城首页是用户进入系统后的第一个页面，承担着商品展示和导航引导的核心功能。页面布局从上到下依次包含顶部导航栏、商品分类导航和商品列表展示区三个部分。

顶部导航栏固定在页面顶部，提供全局导航功能。左侧显示系统Logo，点击可返回首页。中间提供搜索框，用户可以输入关键词搜索商品，搜索结果会跳转到商品列表页面。右侧提供购物车入口和用户中心入口，购物车入口显示当前购物车商品数量，用户中心入口提供用户登录、注册、个人信息管理和订单查看等功能。

商品分类导航横向展示商品分类列表，用户点击分类名称可以快速筛选该分类下的商品。分类数据从后端接口获取，支持多级分类展示。

商品列表展示区以网格形式展示商品卡片，每个卡片包含商品图片、名称、价格和加入购物车按钮。商品列表支持分页加载，用户滚动到底部时自动加载下一页商品。列表支持按价格、销量、上架时间等维度排序，用户可以选择最适合自己的排序方式。

**（2）商品详情页**

商品详情页展示单个商品的完整信息，帮助用户做出购买决策。页面布局包含商品图片区、信息区和操作区三个部分。

商品图片区展示商品的主图和缩略图，用户点击缩略图可以切换主图显示，支持图片放大查看功能。多角度的图片展示帮助用户全面了解商品外观。

价格和库存信息区域突出显示商品价格和库存状态。价格以大字号显示，库存状态用文字提示（如"有货"、"缺货"）或图标表示。如果库存紧张，系统会提示用户尽快下单。

操作区域提供加入购物车和立即购买两个主要按钮。加入购物车按钮将商品添加到用户购物车，返回购物车页面可以继续购物。立即购买按钮跳转到确认订单页面，可以快速完成购买流程。这两个操作都需要用户登录后才能执行，未登录用户会提示先登录。

商品详细描述区域使用富文本展示商品的详细信息，包括商品特点、规格参数、使用方法等。用户可以滚动查看完整描述。

**（3）购物车页面**

购物车页面管理用户的购物车商品，是用户下单前的重要环节。页面布局包含商品列表区和结算区两个部分。

商品列表区以列表形式展示购物车中的商品，每个商品项包含商品图片、名称、单价、数量调整控件和删除按钮。数量调整控件支持手动输入和加减按钮操作，最小数量为1，最大数量受库存限制。删除按钮提供单个商品删除功能，删除前会弹出确认提示。

每件商品下方显示该商品的小计金额，计算方式为单价乘以数量。列表顶部显示已选商品数量。

结算区域固定在页面底部或右侧（视页面宽度而定），实时显示购物车商品的总金额。金额计算包括商品小计总和，用户可以看到清晰的费用明细。结算按钮触发跳转到订单确认页面。

**（4）智能助手页面**

智能助手页面提供智能购物咨询功能，帮助用户找到合适的商品。页面布局采用左侧会话列表、中间对话区域、底部输入框的标准聊天界面设计。

左侧会话列表展示用户的历史会话记录，每个会话项显示会话标题、创建时间和最后一条消息的预览。点击会话可以加载该会话的历史消息继续对话。提供新建会话功能，用户可以开始一个全新的对话。

中间对话区域显示当前会话的消息列表，用户消息和助手消息以不同样式区分显示。用户消息通常靠右显示，助手消息靠左显示。助手消息下方会展示商品推荐卡片，如果助手在回复中推荐了商品，卡片会包含商品图片、名称、价格和查看详情按钮。

底部输入框提供文本输入和发送功能。用户输入消息后点击发送按钮或按回车键发送消息。发送时输入框清空，用户消息立即显示在对话区域，助手消息区域显示加载状态指示器。如果加载状态时间过长，系统会显示超时提示。

### 5.8.2 日志与异常处理

系统实现过程中，日志与异常处理是保障运行稳定性的关键环节。后端使用Slf4j作为日志框架，这是Java生态中最常用的日志门面系统，提供了统一的日志API，支持多种日志实现（如Logback、Log4j2）。

后端日志记录覆盖关键业务流程的各个环节。用户登录时记录用户ID和登录结果，便于追踪用户行为和排查登录异常。订单创建时记录订单号、用户ID和订单金额，便于订单问题排查和财务对账。LLM调用时记录请求内容、响应状态和耗时，便于性能监控和问题诊断。当LLM调用失败时，会记录详细的异常堆栈信息，便于定位问题原因。

日志级别根据情况选择合适的级别：Info级别记录正常的业务流程，如接口调用、业务处理完成等；Warn级别记录可能存在问题但不影响功能的情况，如库存不足建议补货；Error级别记录系统异常和错误，如数据库连接失败、API调用超时等。

前端使用Element Plus提供的Message组件实现统一的错误提示。请求失败时显示错误消息，如"网络错误，请检查网络连接"。认证失败时显示登录过期提示，并自动跳转到登录页面。业务错误时显示后端返回的错误信息，如"商品库存不足"。

后端实现全局异常处理器，使用@RestControllerAdvice注解标记。该处理器统一捕获业务异常（如用户名已存在、商品不存在）、参数校验异常（如必填参数缺失、参数格式错误）和系统异常（如空指针异常、类型转换异常），将异常信息封装为规范化的JSON响应返回给前端。响应包含错误码和错误消息，便于前端进行针对性处理。

对于智能助手模块，还额外处理了LLM服务特有的异常情况。API超时异常捕获后返回友好提示"服务响应超时，请稍后重试"。返回格式异常捕获后记录错误日志并返回"服务数据异常"提示。限流异常捕获后返回"服务繁忙，请稍后重试"提示。

### 5.8.3 异常处理与日志设计

本系统通过结构化日志记录关键业务事件，确保异常发生时可以快速定位问题根源。日志设计遵循以下原则：

日志内容包含足够的上下文信息，如用户ID、订单ID、商品ID等，便于关联查询相关数据。日志时间精确到毫秒，便于分析事件发生的先后顺序和性能问题。日志格式采用结构化格式（如JSON），便于日志收集和检索系统处理。

后端日志可以与日志聚合平台对接，实现集中式的日志管理和分析。常用的方案包括ELK（Elasticsearch、Logstash、Kibana）技术栈或云服务商提供的日志服务。通过日志聚合平台，开发人员可以快速检索、筛选和分析海量日志数据，大大提高了问题定位效率。

日志记录同时包含正常流程和异常流程的记录。正常流程记录有助于了解系统的运行状况和用户行为模式，异常流程记录有助于排查问题和优化系统。

前端在请求失败时统一提示用户，提示内容根据错误类型有所区别。网络错误提示"网络连接失败，请检查网络设置"。认证失败提示"登录已过期，请重新登录"并自动跳转登录页。服务器错误提示"服务器异常，请稍后重试"。限流错误提示"操作过于频繁，请稍后再试"。

对于智能助手会话，系统特别记录生成失败、超时和重试行为。这些日志数据对于分析模型调用稳定性、系统表现和用户体验至关重要。开发人员可以通过分析这些日志识别LLM服务的常见问题，优化调用策略，提高服务质量。

### 5.8.4 工程实践要点

在实际实现中，系统遵循了前后端分离的设计原则，保持了后端服务与前端页面的功能边界清晰。这种设计提高了系统的可维护性和可扩展性，使得前后端可以独立开发和部署。

后端主要负责数据处理、业务校验和外部服务集成。数据处理包括数据库CRUD操作、数据转换和聚合计算等。业务校验包括参数合法性检查、权限验证和业务规则验证等。外部服务集成包括LLM服务调用、第三方支付对接等。这种职责划分使得后端专注于业务逻辑和数据处理，提供了稳定可靠的API服务。

前端主要负责界面展示、用户输入和请求流转。界面展示包括组件渲染、动画效果和响应式布局等。用户输入包括表单处理、事件响应和状态管理等。请求流转包括API调用、数据缓存和错误处理等。这种职责划分使得前端专注于用户体验和交互逻辑，提供了良好的用户界面。

系统引入了自动化测试、代码审查和持续集成流程，从多个层面保障代码质量和系统稳定性。

自动化测试包括单元测试和集成测试两个层次。单元测试针对单个类或方法的逻辑进行验证，确保每个代码单元的行为符合预期。集成测试针对多个组件的协作进行验证，确保系统各部分能够正确交互。测试覆盖了核心业务逻辑，包括用户认证、商品查询、订单创建和智能助手对话等关键功能。

代码审查通过Pull Request流程实现，团队成员相互审查彼此的代码变更。代码审查可以发现潜在的bug、优化代码结构、分享技术经验，并确保代码符合团队规范。审查重点包括业务逻辑正确性、异常处理完备性、安全性检查和性能考虑等。

持续集成通过自动化流水线实现，每次代码提交会自动触发构建和测试流程。流水线包括代码编译、依赖安装、测试执行、代码检查和构建产物生成等步骤。如果某个步骤失败，团队成员会收到通知并及时修复问题。这种机制确保了代码质量的持续监控和问题早发现、早解决。

## 5.9 本章小结

本章详细介绍了商城智能购物助手系统的实现过程。首先介绍了开发环境的配置，包括后端Maven依赖的组成和作用、前端Vite配置的各个选项含义、以及数据库表结构的设计。然后按模块详细展示了核心功能的实现：用户认证模块通过JwtUtil工具类、JwtInterceptor拦截器和UserServiceImpl服务类实现了JWT令牌的生成验证和用户身份认证；商品模块通过ProductController和ProductServiceImpl实现了商品列表分页查询、分类筛选、关键词搜索和详情查看功能；购物车模块通过CartController和CartServiceImpl实现了购物车商品的添加、查询、数量修改和删除功能，同时实现了库存校验保证数据准确性；订单模块通过OrderController和OrderServiceImpl实现了订单创建的完整流程，包括购物车商品查询、库存验证、金额计算、订单记录创建、订单项记录创建、库存扣减和购物车清空等步骤，通过事务管理确保数据一致性；智能助手模块通过RAGService实现检索增强生成功能，通过LLMService和QwenLLMServiceImpl实现与大语言模型的交互，通过AssistantController和AssistantServiceImpl实现流式对话和商品推荐功能。最后介绍了前端的Axios封装实现统一请求处理和错误管理，SSE客户端实现流式消息接收，智能助手页面组件实现完整的对话交互界面。

通过本章的实现，系统完成了从设计到功能实现的转化，具备了完整的商城智能购物助手功能，包括用户注册登录、商品浏览搜索、购物车管理、订单创建和智能助手对话等核心功能。同时，系统在实现过程中注重开发环境配置、依赖版本管理、代码编写规范和异常处理机制等方面，为后续系统稳定运行和迭代升级提供了坚实基础，使项目不仅在功能上可用，也具备良好的代码质量、可维护性和可扩展性。
