# 任务书：商城智能助手系统（基于大语言模型与RAG）

## 一、设计任务

### 1. 课题名称
商城智能助手系统（基于大语言模型与RAG的电商平台）

### 2. 设计目标
在前后端分离的电商系统基础上集成智能助手能力，实现“业务交易闭环 + 对话式咨询推荐闭环”。系统应支持真实的业务数据检索与可演示的智能交互体验，并具备可扩展升级到向量语义检索的工程结构。

### 2.1 关键成果定位（一句话）
面向电商场景构建“可运行、可演示、可扩展”的智能购物助手系统：既能完成交易闭环，又能通过RAG+SSE提供对话式咨询与推荐服务。

### 3. 主要设计内容与工作任务

#### 3.1 电商业务模块（基础闭环）
- **用户模块**：注册、登录、获取当前用户信息；JWT鉴权；密码BCrypt加密存储。
- **商品模块**：商品列表/详情/搜索（分页与筛选能力可扩展）。
- **购物车模块**：添加、列表查询、数量更新（含库存校验）、删除、清空。
- **订单模块**：创建订单（从购物车结算、扣减库存）、支付（模拟）、取消（恢复库存）、订单列表与详情。
- **（可选扩展）地址模块**：地址新增/修改/删除/设默认；下单选择地址。

#### 3.2 智能助手模块（核心特色）
- **对话与会话管理**：会话列表（分页）、历史消息查询、删除会话；支持续聊（sessionId）。
- **流式输出（SSE）**：实现服务端逐段推送与前端增量渲染，保证首字快速可见。
- **RAG检索与商品推荐**：
  - 关键词/规则检索：从商品数据中召回Top-K候选商品。
  - 将检索结果注入Prompt，生成面向用户的解释型回答。
  - 在对话结束事件中返回推荐商品列表，前端以商品卡片展示并支持跳转详情页。
- **多数据源能力（按项目实现口径）**：支持注册不同数据源类型（商品、订单、用户等），支持单源/多源检索接口设计。

#### 3.3 核心接口与数据对象（用于任务边界与验收）
> 说明：以下为任务书层面的“必须具备/建议具备”的接口与数据对象清单，便于验收时逐项核对。接口路径以项目约定的 `/api/...` 为准。

##### 3.3.1 关键接口（建议清单）
- **用户认证**
  - `POST /api/user/register`：注册
  - `POST /api/user/login`：登录（返回JWT）
  - `GET /api/user/info`：获取当前用户信息（需鉴权）
- **购物车**
  - `POST /api/cart/add`：添加商品到购物车（同商品合并）
  - `GET /api/cart/list`：购物车列表（包含商品详情）
  - `PUT /api/cart/{cartId}`：更新数量（校验库存）
  - `DELETE /api/cart/{cartId}`：删除条目
  - `DELETE /api/cart/clear`：清空购物车
- **订单**
  - `POST /api/order/create`：从购物车结算创建订单（扣减库存）
  - `GET /api/order/list`：订单列表（支持按状态筛选）
  - `GET /api/order/{orderId}`：订单详情
  - `PUT /api/order/{orderId}/pay`：支付（模拟）
  - `PUT /api/order/{orderId}/cancel`：取消（恢复库存）
- **智能助手**
  - `POST /api/assistant/chat/stream`：SSE流式对话（结束返回推荐商品事件）
  - `GET /api/assistant/conversations`：会话列表（分页）
  - `GET /api/assistant/conversation/{sessionId}/messages`：会话历史消息
  - `DELETE /api/assistant/conversation/{sessionId}`：删除会话与消息

##### 3.3.2 关键数据表/实体（建议清单）
- **业务表**：`user`、`product`、`category`、`cart`、`order`、`order_item`
- **对话表**：`conversation`、`conversation_message`
- **（可选扩展）地址表**：`address`

#### 3.3 测试与文档
- **功能测试**：覆盖登录鉴权、购物车、下单/支付/取消、智能助手流式对话与推荐、会话管理等主路径与异常路径。
- **基础性能验证**：普通接口响应与并发基本验证；智能助手首字响应与稳定性验证。
- **文档输出**：系统设计说明、接口说明、部署运行说明、测试报告（用于论文整合与答辩材料）。

### 3.4 里程碑交付物（每阶段“看得见”的产出）

## 二、设计要求

### 1. 功能性要求
- **鉴权**：需要登录的接口必须通过JWT校验；未授权访问返回统一错误码。
- **业务正确性**：
  - 购物车合并同类商品，数量更新需校验库存。
  - 创建订单需校验库存并正确扣减；取消订单应恢复库存；订单状态流转正确。
- **智能助手**：
  - 支持流式对话；对话结束可返回推荐商品列表。
  - 支持会话列表与历史消息查询、删除会话等基础管理能力。
  - RAG检索结果应与回答内容保持一致，尽量避免“与库存/价格无关的编造描述”。

### 2. 性能与体验要求
- **页面体验**：主要页面加载与交互流程清晰；关键操作有明确反馈。
- **接口响应**：常规CRUD接口保持稳定快速；智能助手采用SSE确保“首字尽快呈现”。
- **并发与稳定性**：在基本并发场景下系统可用；对话接口出现超时/限流时应返回可理解的降级提示。

### 2.1 建议量化指标（可按学校要求调整）
- **普通接口响应**：核心列表/详情/增删改在常规数据量下响应稳定（以接口测试结果为准）。
- **智能助手体验**：SSE首字尽快呈现；对话结束必须输出推荐事件与done事件；断连/超时有降级提示。
- **稳定性**：重复请求、异常输入不导致服务崩溃；关键异常能返回统一响应结构与可读错误信息。

### 3. 安全性要求
- **密码安全**：密码必须使用BCrypt等强哈希存储，不得明文保存。
- **配置安全**：数据库密码、大模型API Key等敏感信息不得硬编码；避免在日志中输出完整token与密钥。
- **越权控制**：用户仅能访问本人的订单、会话与消息数据。
- **输入校验**：对关键入参进行校验，防止注入与非法参数导致的系统异常。

### 3.1 安全验收要点（建议）
- **鉴权覆盖**：除注册/登录等公开接口外，其余关键接口均需鉴权。
- **越权测试**：使用A用户token访问B用户订单/会话应返回拒绝结果。
- **敏感信息保护**：日志不输出明文密码、完整token与API Key；配置外部化，避免提交到仓库。

### 4. 可维护性与可扩展性要求
- **分层架构**：后端遵循Controller/Service/Mapper分层；统一响应结构与异常处理。
- **模块化**：LLM服务、RAG检索与数据源应接口抽象，便于切换模型供应商与扩展数据源。
- **检索可演进**：保持“关键词检索→向量检索”的升级路径可行，接口与数据结构可复用。

### 4.1 可扩展方向（建议写入任务边界）
- **检索升级**：Embedding与向量数据库接入（Chroma/Milvus/pgvector），并加入业务过滤与重排。
- **多数据源增强**：在商品外，扩展订单/用户数据源，使助手可回答“订单状态/我的订单”类问题。
- **地址管理完善**：下单选择地址、默认地址策略、地址校验与与订单关联。

### 5. 文档与规范要求
- **文档齐全**：需求、设计、实现说明、测试与部署文档完整一致。
- **可复现**：提供数据库脚本与启动说明，确保他人可按文档复现运行。
- **代码规范**：命名清晰、结构一致、关键逻辑具备必要的错误处理与日志记录（避免叙述性无意义注释）。

### 5.1 文档清单（建议）
- **《系统设计说明》**：总体架构、模块划分、关键流程（JWT/订单/助手）、数据库与接口设计。
- **《接口说明》**：核心接口列表、请求/响应示例、错误码与鉴权说明。
- **《部署运行说明》**：环境要求、配置项、启动步骤、常见问题排查（含SSE代理缓冲注意事项）。
- **《测试报告》**：测试环境、用例列表、结果统计、问题与修复记录。
