**概述与背景**

该商城项目包含前端（基于 Vite + Vue3）、后端（基于 Spring Boot）、及智能助手功能模块。现有工作区中已包括前端组件 `mall-frontend/src/components/assistant` 与后端 `mall-backend/src/main/java/com/mall/service/assistant` 的占位或实现。目标是完善功能，稳定生产部署，并引入基于大模型的智能助手能力提升用户体验与客服效率。


**目标与关键指标（KPI）**
- 提供稳定的商品浏览、搜索、下单与支付流程，月活用户（MAU）可扩展至目标值。
- 关键性能指标：页面首屏加载时间 < 1s（服务端渲染或预缓存场景除外），API 平均响应 < 200ms（一般查询），系统可承载并发请求峰值 N（由部署环境确定）。
- 智能助手：目标回答准确率≥85%（针对商品/订单相关问答），故障回退率 < 1%，平均响应延迟 < 1s（模型推理外网延迟/检索时间另计）。


**用户角色与用例**
- 访客：浏览商品、搜索、查看详情。
- 注册用户：加入购物车、下单、查看订单、管理地址、评价商品、使用智能助手。
- 管理员/运营：商品管理、订单管理、用户管理、统计报表、审核内容。
- 客服人员：在系统后台利用智能助手辅助回复、查看用户会话与订单信息。


**功能性需求**

前端（客户）
- 商品浏览：商品列表分页、分类过滤、排序。
- 搜索：支持关键字搜索、模糊匹配、按类目与价格区间过滤、联想建议。
- 商品详情：展示商品图片、规格、库存、评价、推荐关联商品。

- 个人中心：订单列表、地址管理、收藏、账户信息、登录/注册。
- 智能助手：浮动入口，支持会话历史、消息输入、商品推荐卡片、跳转商品详情、会话内快速下单能力。

国际化与本地化

**大模型（智能助手）详细需求**

1) 目标与能力范围
- 支持商品问答、订单查询、推荐与对话式购物引导。
- 能够解释检索结果来源（traceability）、给出基于证据的回答。

3) 能力边界与限制
- 明确大模型仅能基于已接入的数据源和检索结果进行回答，涉及法律、医疗、金融等高风险内容时需自动拒答或转人工。
- 对于超出知识库范围的问题，需有“无法回答”兜底机制。
- 对于多轮复杂推理、长链路问题，需有分步澄清与上下文压缩策略。

4) 微调与持续学习
- 支持通过有标注的商城问答数据对大模型进行微调（如 LoRA/QLoRA/P-Tuning），提升商品、订单、客服等领域的专业性。
- 建立用户反馈收集机制，将用户纠错、人工补答等数据纳入训练集，定期迭代模型。

5) 上下文窗口与多轮对话
- 明确支持的最大上下文窗口长度（如 8k/16k/32k tokens），超长对话需自动摘要或分段。
- 多轮对话需保存上下文状态，支持话题切换、回溯与澄清。

6) 推理成本与资源管理
- 需统计每次推理的 token 数、时延与费用，支持多模型路由（如高价值问题用大模型，常见问题用小模型或检索）。
- 支持模型调用频率、并发数、超时等参数的动态配置。

7) A/B 测试与模型评估
- 支持多版本模型灰度发布与 A/B 测试，评估不同模型/提示词/检索策略对用户体验的影响。
- 评估指标包括但不限于：准确率、召回率、用户满意度、平均响应时延、成本、人工介入率。

8) 用户体验保障
- 对于模型生成的不确定、模糊或低置信度回答，需主动提示“建议人工咨询”或提供人工客服转接。
- 支持用户对模型回答进行反馈（点赞/点踩/纠错），并可追踪到具体会话与问题。

9) 数据安全与合规性
- 严格遵守《个人信息保护法》《数据安全法》等相关法规，所有涉及用户隐私的数据需加密存储与传输。
- 对模型训练、推理、日志采集等环节进行权限隔离与访问审计。
- 对外部 API/模型服务调用需有 SLA 保障与合规协议。

10) 其他前沿能力建议
- 支持多模态扩展（如商品图片理解、语音输入/输出），为后续升级预留接口。
- 支持知识库热更新与在线学习，提升新商品/新活动的响应速度。

2) 数据架构与 RAG（检索增强生成）数据源
- 优先数据源：商品库（商品表与规格）、订单库（订单表、物流表）、FAQ 文档、帮助中心、站内文档（docs 目录）、用户会话历史。
- 将业务数据抽取为知识片段（文档切片），并建立元数据（source、doc_id、timestamp、schema_type、关联商品 id）。
- 对于频繁更新的数据（如库存/价格），需建立短时缓存与检索刷新策略，避免旧信息误导生成结果。

3) 向量化、索引与檢索
- 文档切片策略：按语义分片（商品描述按段、FAQ 按问答对、订单记录按条）。推荐的切片长度：512–2000 字符，依据语言与向量化模型调整。
- 向量化模型：采用开源或托管的文本嵌入模型（例如 OpenAI/Claude embeddings、SentenceTransformers、Open-Source LLM embeddings）；应支持批量向量化与增量更新。
- 向量库：选择 Milvus/Weaviate/FAISS（托管或自建）作为向量存储，支持近实时插入与检索、向量过滤（metadata filter）。
- 检索策略：先使用向量召回 K（推荐 10–50），再用稀疏检索（如 ElasticSearch）合并/重排。

4) 模型技术选型与部署方案
- 推理模型：可采用小到中等规模的 LLM（例如 Llama-2/3 家族或托管大模型）用于生成，结合 RAG 提高准确性与证据可追溯性。
- 部署方式：优先使用托管 API（云端）以降低运维成本；对于严格的隐私或低延迟需求，考虑自托管推理集群（GPU 节点 + Triton/LLM Server）。
- 缓存策略：对高频问答与推荐结果做短时缓存，避免重复调用模型。

5) 提示工程、对话管理与模板
- 设计分层系统提示：系统级（行为限制）、功能级（检索/生成模板）、场景级（下单、投诉、物流查询）。
- 对于 RAG：在生成 prompt 时包含检索到的证据片段（带来源元数据），并要求模型以“依据来源”方式回答。
- 会话状态管理：保存对话上下文（最近 N 条消息 + 会话元信息），并在必要时压缩历史以避免超长上下文。

6) 评估、监控与可观测性
- 指标：准确率（基于人工标注的一组问题）、召回率@K（检索质量）、答复命中率（是否直接解决用户意图）、平均延迟、资源成本（每次请求花费）。
- 监控：记录模型输入、模型输出、检索证据、延迟、错误率；对敏感/失败用例采样并人工审查。

7) 隐私、合规与成本控制
- 对话中包含 PII（姓名、地址、电话、订单号）时，必须按策略脱敏或仅通过后端安全接口返回最小化信息。
- 建议对外部托管服务使用专有加密传输与访问控制，必要时对敏感数据在发送前做脱敏或摘要化处理。
- 成本控制：设置频率上限、按需扩缩容、对低价值请求采用检索或模板回复代替模型生成。

8) 防止幻觉与安全策略
- 强制模型输出来源引用；对来源不充分或检索不足的情形返回“我不确定，请稍后”或转人工。
- 对生成内容进行安全审查（违规词库、商家违规内容检测）。

**API 与接口契约示例**

1) 用户发起会话/发送消息
- POST /api/assistant/message
  请求体：
  {
    "userId": "string",
    "sessionId": "optional string",
    "message": "string",
    "context": { ... }
  }
  响应：
  {
    "sessionId":"...",
    "reply":"...",
    "evidence":[ {"source":"faq","id":"...","snippet":"..."} ],
    "suggestions":[ {"type":"goto_product","productId":"..."} ]
  }

2) 后端检索接口（内部）
- POST /internal/assistant/retrieve
  参数：{ "query":"...","k":20,"filters":{...} }
  返回：向量检索命中结果及元数据。
